# Terminal inline xterm.js plutôt que spawn OS natif

PDO doit exposer à l'utilisateur le contenu vivant de chaque session tmux Claude Code (NodeRun + Manager). Première itération (CONTEXT.md "option A") : preview read-only via `tmux capture-pane` polled toutes les 1-2 s, et bouton *"Open terminal"* qui exec un terminal OS natif (`gnome-terminal`/`konsole`/`Terminal.app`/`kitty`...) attaché à la session via `tmux attach`. Limite : la fenêtre OS sort du flow de l'app, casse l'expérience "atelier de production" (CONTEXT.md "UX — modes Run et Edit"), et oblige à maintenir un détecteur de terminal cross-platform fragile.

**Décision : émulateur de terminal inline dans l'UI web (xterm.js + addons), backé par un PTY côté daemon qui exécute `tmux attach` et bridge les bytes I/O via WebSocket.** L'utilisateur voit un terminal interactif **dans le panneau de détail du nœud** (et dans l'onglet info de la toolbar pour le Pipeline Manager), avec deux icônes : agrandir (le terminal occupe tout l'espace vertical du panneau) et détacher (spawn d'une fenêtre OS native, comportement legacy gardé en option). Le polling 1-2 s de `capture-pane` disparaît : la WebSocket pousse en temps réel.

**Pourquoi.** Choisi contre l'alternative *"garder l'option A spawn OS natif comme seul mécanisme"* parce que (1) ça maintient l'utilisateur dans l'app sans context-switch fenêtre, ce qui est la condition du principe *Deliberate over autonomous* (l'utilisateur est censé pouvoir intervenir à tout moment, friction = moins d'interventions), (2) ça supprime un détecteur de terminal cross-platform qui est par construction fragile, (3) le coût impl est modéré (`portable-pty` + xterm.js sont des libs matures, ~3-5 jours pour le bridge + l'intégration). Choisi contre l'alternative *"inline only, supprimer complètement le spawn OS"* parce qu'un escape hatch via icône détacher reste utile dans des cas limite (besoin d'un vrai terminal pour une opération de copy/paste exotique, debug d'un freeze de la WebSocket). Conséquences : nouvelle dépendance frontend `@xterm/xterm` + addons (`fit`, `web-links`), nouvelle route `WS /sessions/<id>/pty` côté daemon, nouveau module `pty_bridge.rs`, ajout d'un origin check sur la WebSocket pour éviter le DNS-rebinding (le daemon écoute sur `127.0.0.1` mais ce n'est pas suffisant en soi). Le multi-client par session (deux onglets browser sur la même session tmux) est gratuit côté tmux, à documenter en non-fonctionnel plutôt qu'à coder.
