# Le Merge devient un nœud first-class, plus auto-spawné

Première itération (CONTEXT.md "Merge Resolver") : un rôle `code-mutating` built-in, **auto-spawné** par le runtime quand un `git merge` entre branches `code-mutating` parallèles produit un conflit. Toggle `auto_merge_resolver: enabled | disabled` au niveau pipeline. Cette formulation a deux défauts : (1) elle introduit de l'orchestration ambiante (le runtime décide de spawner sans que le designer l'ait demandé), ce qui contredit *Sharp tool* (ADR-0001) — le designer ne *voit* pas la barrière de synchro dans son DAG ; (2) elle ne couvre que le cas conflit, pas la barrière elle-même : un fan-out parallèle de N branches `code-mutating` n'a aujourd'hui aucun nœud explicite pour rassembler, c'est implicite. Le besoin a explosé avec l'arrivée du nœud `ForEach` (parallèle obligatoire), qui rend la barrière + fusion un pattern *de premier ordre*, plus une exception.

**Décision : le `Merge` devient un nœud first-class du DAG, type `code-mutating` toujours, à placer explicitement par le designer.** Sa forme : 1 input port `branches: repeated` (accumule N edges parallèles), 1 output port `merged` (artefact résumé du merge avec frontmatter `conflict_count`, `branches: [...]`, et corps narratif). Sémantique : (a) attend que toutes les branches upstream soient terminées (barrière intrinsèque, gratuite vu que le scheduler n'évalue le node que quand ses upstream sont tous `Completed`) ; (b) forke un sous-worktree depuis la branche du Pipeline Run ; (c) tente `git merge` sur chaque upstream qui a une branche dédiée (= les `code-mutating`) ; (d) si conflit → spawn Claude Code dans le sous-worktree, qui lit les artefacts du Blackboard pour reconstituer les intentions, résout, commit ; (e) si pas de conflit → écrit un summary trivial *"merge auto, no conflict"* + commit, sans LLM. Le toggle `auto_merge_resolver` disparaît. Le Merge auto-spawné disparaît.

**Pourquoi.** Choisi contre l'alternative *"garder l'auto-spawn et y greffer les barrières ForEach"* parce que ça aurait empilé deux mécanismes implicites (barrière + résoluteur) que le designer ne voit ni ne configure — incompatible avec *Sharp tool*. Choisi contre l'alternative *"deux nœuds distincts `Merge` (code) et `Join` (doc-only)"* parce que la sémantique "rassembler N branches" est unique du point de vue designer : il wire ses N branches vers le port `branches` et le runtime fait le bon choix selon le type des upstream — forcer deux nœuds pour le même geste est friction inutile. Choisi contre l'alternative *"spawn LLM systématique pour traçabilité"* parce que le coût (un Claude Code par convergence, même triviale) écrase le bénéfice ; le summary trivial sans LLM suffit pour l'audit. Conséquences : pipelines existantes qui s'appuyaient sur l'auto-spawn cassent — un script de migration `pipeline_migrator` doit insérer un Merge node downstream de chaque fan-out `code-mutating` détecté, sinon le runtime n'a plus rien pour résoudre les conflits et la branche du Run reste cassée. Diagnostic info-only dans l'éditeur si fan-out `code-mutating` sans Merge downstream (cf. ADR-0001 : lint info-only autorisé).
