# Édition du graphe pendant qu'un Run tourne

CONTEXT.md initial dit *"topologie figée, tracée à l'édition"* (Pipeline) et sépare un **mode Edit** (toggle global, icône crayon) d'un **mode Run** (par défaut). Implication implicite : éditer un graphe et l'exécuter sont deux phases discrètes — on conçoit, puis on lance, puis on attend. Cette dichotomie crée trois frictions : (1) elle force un mental switch alors que la majorité des outils du marché (n8n, Langflow, Dify, ComfyUI) ont un canvas toujours interactif ; (2) elle empêche l'utilisateur d'ajouter une node en cours d'exécution alors que le run vient de produire une info qu'il aurait voulu router différemment — geste cohérent avec le principe *Deliberate over autonomous* (l'utilisateur peut intervenir à tout moment) ; (3) elle invente une distinction "définition vs instance" que le code n'avait déjà pas : chaque Run a son propre snapshot pipeline sous `<repo>/.maestro/runs/<run-id>/`, watché par le pipeline_watcher, et le scheduler est une fonction pure réévaluable à chaque modif (`scheduler::ready_nodes`, `scheduler::evaluate_outgoing_edges_with_context`). La fondation pour l'édition à chaud existe ; le toggle Edit/Run la masque.

**Décision : un seul mode d'édition, qui s'adapte selon l'état du graphe (running ou pas).** Le canvas est toujours interactif. Quand un Run tourne, l'édition modifie le snapshot run-scope (`<repo>/.maestro/runs/<run-id>/pipeline.yaml`) ET propage la même modif vers la template d'origine en bibliothèque (auto-sync montant). Quand aucun Run ne tourne sur une pipeline, l'édition modifie directement la template en bibliothèque. **Politique de mutation pendant un Run** : (a) interdiction stricte de supprimer un node de status non-`pending` ; (b) édition autorisée du `max_iter` d'un Loop live (équivaut à la commande `extend_cycle` du Pipeline Manager, qui devient redondante) ; (c) ajout de node + edge libre — si la nouvelle edge active un node non-encore-spawné, le scheduler le pickup au prochain tick ; les nodes already-completed/running ne re-tournent pas, modif sans effet runtime sur leur iter en cours mais visible à l'iter suivante (Loop). **Étanchéité entre runs** : modif d'un run-snapshot n'impacte aucun autre run en cours (chaque run a son propre snapshot). Modif d'une template hors-Run n'impacte aucun run en cours (qui ont leur propre snapshot). L'auto-sync montant ne va que du run-scope vers la template, jamais l'inverse.

**Pourquoi.** Choisi contre l'alternative *"modèle draft + published à la n8n/Zapier/Prefect"* (édition modifie un draft, run en cours figé sur sa version snapshot, modifs s'appliquent au prochain run) parce que (1) Maestro est mono-user local — il n'y a pas de collègue qui exécute un workflow en prod pendant que tu édites en draft, le risque de surprise mutuelle est nul ; (2) la motivation principale d'un draft (audit, reproductibilité) est déjà couverte par l'event log SQLite append-only, qui contient toute la séquence d'événements d'un Run ; (3) le principe *Deliberate over autonomous* prépare exactement ce geste — l'utilisateur attache une session, voit un output qu'il n'avait pas anticipé, ajoute un Reviewer downstream à la volée, le scheduler le pickup. C'est sciemment hors-marché : aucun outil mainstream ne fait du hot-rerouting d'un graphe en cours d'exécution. Choisi contre l'alternative *"snapshot run-scope sans auto-sync vers la template"* (modifs run-locales par défaut, promotion explicite via bouton) parce que ça force un geste supplémentaire alors que le cas dominant est *"je débugge ma template via un run, je veux que ma correction colle"*. La friction se produirait sur le cas inverse (je veux essayer un patch jetable sans polluer la template) ; on l'addresse plus tard si un cas concret le justifie, par exemple via un toggle "this run only" sur l'édition. Conséquences : le toggle Edit/Run global de l'UI disparaît, *Sharp tool* (ADR-0001) reste cohérent puisqu'on n'ajoute aucune validation prescriptive à l'édition à chaud — les seules garde-fous sont des invariants de cohérence runtime (un node `running` ne peut pas être supprimé pour ne pas orphaner sa session tmux). À documenter dans CONTEXT.md : la formulation *"topologie figée"* dans la section Pipeline doit être réécrite pour préciser qu'elle parle du **déterminisme d'orchestration** (pas de LLM-router probabiliste), pas de l'immutabilité du graphe pendant l'exécution.
