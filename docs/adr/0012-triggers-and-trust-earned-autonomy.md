# Triggers + autonomie gagnée par la confiance (révise « pas d'auto-merge, jamais »)

Jusqu'ici, lancer un Run exigeait un geste humain (modale New Run) et un input non-vide, et le principe *Deliberate over autonomous* posait un interdit dur : « **Pas d'auto-merge vers main, jamais.** L'humain tranche les actions à effets durables. » Le besoin (#104 scheduled/script-triggered runs, #137 pipelines runnable sans prompt) est d'automatiser des Runs récurrents — un process auditeur qui crée des issues `ready-for-agent`, un fixer qui les ramasse sur cron ou à leur création — donc d'aller, à terme, jusqu'au push/PR/merge sans intervention. Cela entre frontalement en conflit avec le « jamais ».

**Décision.** On introduit le **Trigger** comme nouvelle entité first-class, et on **révise le principe** : *Deliberate, then autonomous (trust-earned)*. Le défaut reste délibéré (humain dans la boucle) ; l'autonomie devient une **cible atteignable, pas un interdit**.

(a) **L'autonomie est une propriété du pipeline, jamais du runtime ni du Trigger.** Le « jamais » ne porte plus sur l'*effet* (merge, PR, cleanup) mais sur son *origine* : le runtime ne déclenche jamais d'action durable de lui-même ; si elle se produit, c'est qu'un **nœud du pipeline** l'exécute (Shipper avec `gh pr create`, nœud de merge), choix explicite du designer, versionné et auditable. Conséquence : un pipeline auto-shippant se comporte à l'identique lancé à la main ou par Trigger — pas de divergence manuel/automatique. La confiance s'audite sur le pipeline.

(b) **Un Trigger est start-only** : il lie une condition de déclenchement à un template de Run (= charge d'un `POST /runs`) et ne décide jamais de la terminaison. Condition = **heartbeat cron + guard script optionnel** (`sh -c` dans le repo cible, exit 0 fire / stdout = input, timeout 60 s hors tick). **Un firing = un Run** (la multiplicité se gère via une boucle `collection` dans le pipeline). **Idempotence déléguée au monde extérieur** : Maestro ne tient aucun `fired_keys` ; le pipeline mute l'état qu'il poll (label GitHub = registre de dedup). Safety : **skip de recouvrement par-Trigger** + **cap global de sessions NodeRun** avec attente *au niveau nœud* (managers exemptés). Persistence en tables SQLite `triggers` + `trigger_fires` (audit). Provenance `triggered_by` sur le Run. Flag pipeline **`prompt_required`** (#137, défaut `true`) pour rendre le prompt optionnel. Détails mécaniques complets : CONTEXT.md → *Trigger*, *prompt_required*, *Cap de sessions concurrentes*.

**Pourquoi.** Contre *une termination-policy sur le Trigger* (cases « auto-PR / auto-merge » appliquées par Maestro après le Run) : ça scinde le comportement manuel/déclenché, duplique ce qu'un nœud Shipper fait déjà, et place les actions effrayantes hors du graphe auditable. Mettre l'autonomie *dans le pipeline* la rend composable et inspectable — c'est là que se construit la confiance. Contre *un script-trigger complet où Maestro fan-out N Runs et tient le dedup* : sur-ingénierie — la boucle `collection` gère déjà la multiplicité, et un moteur de dedup *devinerait* l'intention ; on préfère le guard cheap + dedup externe (*Sharp tool*, ADR-0001), au prix assumé qu'un guard mal configuré + pipeline qui ne mute rien boucle (borné par le skip). Contre *cron seul* (polling dans un pipeline self-sufficient déclenché au timer) : chaque tick spawnerait un Run complet juste pour découvrir « rien à faire », gaspillant des sessions (cf. tmux-collapse #78) et polluant la liste — le guard gate *avant* le spawn. Contre *back-pressure seule sans skip* : le cap empêche l'effondrement tmux mais pas l'empilement sémantique (re-fire sur la même issue #42 pendant que le fixer la traite) ; le skip ferme cette fenêtre. Contre *YAML sur disque pour les Triggers* : ce sont config + état de scheduling mutable (réécrit à chaque tick), pas un artefact canvas-backed — la table SQLite est le bon fit et n'entame pas l'event-sourcing (qui reste la vérité du *Run*).

Risques / limites acceptés : daemon **best-effort** v1 (les Triggers ne firent que tant que le process vit ; service persistant systemd/launchd → #156) ; fires manqués **forward-only** (correct par construction — un poll forward voit tout le travail accumulé) ; imbrication/chaînage de Triggers exclue (les pipelines se couplent par le monde extérieur, jamais par un wiring interne). Révise le « jamais » de *Deliberate over autonomous* ; interagit avec ADR-0006 (Shipper/Merge restent des nœuds first-class — le lieu de l'autonomie) et ADR-0001 (*Sharp tool* — l'idempotence est à l'utilisateur).
