Maestro — tmux session & memory audit

How sessions are spawned and reaped, where memory/slot creep still comes from, and when to start (or not start) parallel runs.
Date 2026-06-10 Codebase version 0.1.13 Scope maestro-daemon session lifecycle Method code read + live issue state
TL;DR

The raw resource problem I had on file is largely solved: there is now a global session cap (default 10), a periodic orphan reaper, and the test-leak that burned real Claude sessions is shipped-fixed. What replaced it is quieter and more important to your question:

  • The creep is no longer OOM — it's slot starvation. A leaked or zombie session still counts as Running in the DB, so it permanently occupies one of your admission slots (cap default 20). Fill the cap with zombies and the daemon admits nothing: every new run hangs in waiting forever, looking idle while actually being wedged.
  • The daemon still cannot tell when tmux dies underneath it (issue #77, closed not-planned, never implemented). That's the main zombie source for crashes/logout.
  • Six open scheduler bugs (#194–#199) actively manufacture zombies on otherwise-healthy runs: re-spawn onto non-existent inputs, self-edge stalls, double-spawn races. These are the live creep generators today.

1 · Two failure families

Almost everything in my notes about "memory issues with tmux sessions" collapses into two distinct families. They have different causes, different fixes, and only one of them is mostly handled.

A · Resource exhaustion mostly fixed

Too many live Claude Code instances at once → tmux/gnome-terminal server collapses (observed once: ~30 tmux-spawn-*.scope units, peaks to 6.4 GB on a single scope, server died at 00:22).

Bounded now by the admission cap. Residual risk: no per-session memory ceiling, manager sessions uncapped.

B · Bookkeeping desync (zombies) live problem

The DB says a node is running but no tmux session backs it. Causes: tmux died (#77), or the scheduler re-spawned / double-spawned a node onto an input that never existed (#194–#199).

Each zombie holds a slot forever and burns Claude tokens if its session is actually alive but orphaned. This is your current pain.

3 · The real creep mechanism today

This is the non-obvious part and the answer to "why does it feel like memory keeps creeping even with a cap."

The cap counts live sessions from projected DB state, not from tmux reality:

// admission.rs — a node "holds a session" purely by its DB status
fn node_holds_session(status) -> bool {
    matches!(status, NodeStatus::Running | NodeStatus::AwaitingUser)
}

Nothing in the daemon ever moves a node out of Running when its tmux session vanishes. The reaper only walks tmux → DB (kill sessions whose run is archived/gone); it never walks DB → tmux (flag nodes whose session is gone). So:

The wedge
Every zombie node permanently occupies one admission slot. Slots never free for zombies. After enough accumulate (tmux crash, scheduler re-spawn bug, mid-run node-type edit), your effective cap (default 20) shrinks toward zero. New runs spawn, every node lands in waiting, and the system looks idle while being completely wedged. No error, no OOM — just silent starvation.

Two corollaries worth internalising:

  • A zombie whose tmux is actually still alive (orphaned by a scheduler bug, e.g. re-spawned onto a missing iter-N input per #194/#195/#199) keeps a real claude process running and burning tokens — on a run the UI already shows as completed.
  • Restarting the daemon does not clear DB zombies — status lives in the event log. The reaper's TTL only helps once the run is archived. Until then the slot stays burnt.

5 · Operational playbook — when to start sessions

You want to keep running multiple sessions. The rule isn't "fewer runs" — it's "don't let zombies eat your slots, and respect the real memory budget the cap doesn't enforce."

✓ Start a new run when…

  • Live node sessions are well under the cap (20). Check first (see §6). The cap will queue you, but a queued node behind zombies waits forever.
  • You have RAM headroom. Budget ~1.5 GB per concurrent node session + ~1 GB per manager. free -h before launching 3+ heavy runs (simple-bugfix, sandcastle waves).
  • The work is genuinely independent — different pipelines / unrelated issues. Parallelism pays here.
  • No self-edges in the pipeline (until #194 ships). A self-edge guarantees a stall.

✗ Don't start / don't parallelise when…

  • Zombies are present. Clear them first (§6) or you're launching into a shrunk cap.
  • You'd exceed ~10–12 GB of Claude sessions. The cap won't stop you from OOMing the box once managers + your own dev Claude are added.
  • The pipeline has a bounded loop with a non-loop feeder upstream — #199 will drag the feeder into laps and leak sessions. Run it solo and watch it.
  • You're tempted to edit the pipeline mid-run to "fix" a stuck node. Editing node type breaks completion. Stop the run instead.
  • Just to "resume" a run whose tmux died. Resume re-iterates nodes (#195). Use cleanup_run and relaunch.
Sessions that are not useful
  • Manager sessions on finished runs — they linger until the run is archived/reaped (up to TTL). Archive or cleanup_run finished runs promptly to release them.
  • Interactive (AwaitingUser) nodes you've walked away from — they hold a slot indefinitely by design. If you're not coming back, stop the run.
  • Any node on a run the UI shows completed/failed but that still has live tmux — pure waste (scheduler-bug zombie). Kill on sight.

6 · Detect & recover

Detect zombies (DB says running, tmux says no)

This is the check the daemon doesn't do for you. Run it before launching parallel work:

# 1. How many slots does the DB think are in use? (Running / AwaitingUser nodes)
#    via the daemon API — adjust port (live = 6172 debug here):
curl -s localhost:6172/runs | jq '[.[] | select(.status=="running")]'

# 2. What tmux sessions actually exist?
tmux ls 2>/dev/null | grep '^maestro-'

# 3. Cross-check: a node "running" in (1) with no matching session in (2) is a ZOMBIE.
#    Session names are maestro-<run_id>-<node_id>-iter-<n> ; managers are maestro-mgr-<run_id>.

# 4. RAM reality check before parallelising:
free -h

Recover

  • Zombie / stuck-running node: POST /runs/{id}/commands {"kind":"cleanup_run"} — releases the slot and destroys the worktree. Do not try to "resume".
  • Orphaned live tmux on a finished run: tmux kill-session -t <name>, then archive/cleanup the run so the DB slot frees too.
  • Whole tmux server collapsed mid-run: every affected node is now a zombie. cleanup_run each, then relaunch from scratch. Restarting the daemon alone won't fix the DB status.
  • "git add failed in sub-worktree" after a mid-run type edit: recreate the expected sub-worktree at the snapshot commit so the no-op merge succeeds, then re-run maestro complete — or just stop and relaunch.

7 · Recommendations

Product (in priority order)

  1. Reopen the spirit of #77: add a DB→tmux liveness sweep. Single highest-leverage fix. Periodically, for every node the DB marks Running/AwaitingUser, check tmux has-session; if gone, emit node_stale/run_halted so the slot frees and the UI tells the truth. This directly stops slot starvation. (The #194 plan already proposes a no-progress detector via a node_done epilogue + stale sweep — fold this in.)
  2. Ship the #194–#199 cluster. They're the active zombie factory on healthy runs. #194 (self-edge / positional-iter) and #199 (lap past max_iter dragging in feeders) are the worst leakers. ADR-0013 is the planned home for the resolution-semantics rewrite.
  3. Cap or count manager sessions, or give them a tiny memory ceiling. They're "light" individually but unbounded across parallel runs.
  4. Delete dead scheduler code (iter_for_node, materialize_missing_regions, scheduler_step, TickQueue). It actively misleads anyone reasoning about iteration — including future-you and agents working #194.
  5. Sub-worktree GC on node completion / run archive — bounds disk and inotify.
  6. Optional: pre-spawn free -h guard — refuse/queue admission below a memory floor, since the count cap doesn't model memory.

Operational (today, no code)

  • Drop MAESTRO_SESSION_CAP to 5 for your dev box if you run heavy pipelines alongside your own work.
  • Run the §6 zombie cross-check before any parallel launch. If you do this twice a week you'll never hit silent starvation.
  • cleanup_run finished runs promptly — don't let manager sessions and slots linger.
  • Never edit a running pipeline; never "resume" a tmux-dead run.
  • Keep run 9c8d123 until #194–#199 land (it's the repro), then cleanup_run it.
One-line mental model

The cap stops you from melting the machine. It does nothing about zombies eating your slots — and zombies are now the dominant failure mode. Watch the gap between "DB says running" and "tmux actually exists," and you've covered 90% of the creep.
