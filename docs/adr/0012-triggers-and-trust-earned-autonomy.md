# Triggers + autonomie gagnée par la confiance (révise « pas d'auto-merge, jamais »)

Jusqu'ici, lancer un Run exigeait un geste humain (modale New Run) et un input non-vide, et le principe *Deliberate over autonomous* posait un interdit dur : « **Pas d'auto-merge vers main, jamais.** L'humain tranche les actions à effets durables. » Le besoin (#104 scheduled/script-triggered runs, #137 pipelines runnable sans prompt) est d'automatiser des Runs récurrents — un process auditeur qui crée des issues `ready-for-agent`, un fixer qui les ramasse sur cron ou à leur création — donc d'aller, à terme, jusqu'au push/PR/merge sans intervention. Cela entre frontalement en conflit avec le « jamais ».

**Décision.** On introduit le **Trigger** comme nouvelle entité first-class, et on **révise le principe** : *Deliberate, then autonomous (trust-earned)*. Le défaut reste délibéré (humain dans la boucle) ; l'autonomie devient une **cible atteignable, pas un interdit**.

(a) **L'autonomie est une propriété du pipeline, jamais du runtime ni du Trigger.** Le « jamais » ne porte plus sur l'*effet* (merge, PR, cleanup) mais sur son *origine* : le runtime ne déclenche jamais d'action durable de lui-même ; si elle se produit, c'est qu'un **nœud du pipeline** l'exécute (Shipper avec `gh pr create`, nœud de merge), choix explicite du designer, versionné et auditable. Conséquence : un pipeline auto-shippant se comporte à l'identique lancé à la main ou par Trigger — pas de divergence manuel/automatique. La confiance s'audite sur le pipeline.

(b) **Un Trigger est start-only** : il lie une condition de déclenchement à un template de Run (= charge d'un `POST /runs`) et ne décide jamais de la terminaison. Condition = **heartbeat cron + guard script optionnel** (`sh -c` dans le repo cible, exit 0 fire / stdout = input, timeout 60 s hors tick). **Un firing = un Run** (la multiplicité se gère via une boucle `collection` dans le pipeline). **Idempotence déléguée au monde extérieur** : PDO ne tient aucun `fired_keys` ; le pipeline mute l'état qu'il poll (label GitHub = registre de dedup). Safety : **skip de recouvrement par-Trigger** + **cap global de sessions NodeRun** avec attente *au niveau nœud* (managers exemptés). Persistence en tables SQLite `triggers` + `trigger_fires` (audit). Provenance `triggered_by` sur le Run. Flag pipeline **`prompt_required`** (#137, défaut `true`) pour rendre le prompt optionnel. Détails mécaniques complets : CONTEXT.md → *Trigger*, *prompt_required*, *Cap de sessions concurrentes*.

**Pourquoi.** Contre *une termination-policy sur le Trigger* (cases « auto-PR / auto-merge » appliquées par PDO après le Run) : ça scinde le comportement manuel/déclenché, duplique ce qu'un nœud Shipper fait déjà, et place les actions effrayantes hors du graphe auditable. Mettre l'autonomie *dans le pipeline* la rend composable et inspectable — c'est là que se construit la confiance. Contre *un script-trigger complet où PDO fan-out N Runs et tient le dedup* : sur-ingénierie — la boucle `collection` gère déjà la multiplicité, et un moteur de dedup *devinerait* l'intention ; on préfère le guard cheap + dedup externe (*Sharp tool*, ADR-0001), au prix assumé qu'un guard mal configuré + pipeline qui ne mute rien boucle (borné par le skip). Contre *cron seul* (polling dans un pipeline self-sufficient déclenché au timer) : chaque tick spawnerait un Run complet juste pour découvrir « rien à faire », gaspillant des sessions (cf. tmux-collapse #78) et polluant la liste — le guard gate *avant* le spawn. Contre *back-pressure seule sans skip* : le cap empêche l'effondrement tmux mais pas l'empilement sémantique (re-fire sur la même issue #42 pendant que le fixer la traite) ; le skip ferme cette fenêtre. Contre *YAML sur disque pour les Triggers* : ce sont config + état de scheduling mutable (réécrit à chaque tick), pas un artefact canvas-backed — la table SQLite est le bon fit et n'entame pas l'event-sourcing (qui reste la vérité du *Run*).

Risques / limites acceptés : daemon **best-effort** v1 (les Triggers ne firent que tant que le process vit ; service persistant systemd/launchd → #156) ; fires manqués **forward-only** (correct par construction — un poll forward voit tout le travail accumulé) ; imbrication/chaînage de Triggers exclue (les pipelines se couplent par le monde extérieur, jamais par un wiring interne). `next_fire_at` stocké en **UTC canonique** et comparaison du tick tz-normalisée ; un panic de tick est **isolé** (la boucle survit, fire suivant au prochain tick), liveness exposée par `GET /triggers/health` (#222) ; cron interprété en UTC (heure locale = enhancement séparé). Révise le « jamais » de *Deliberate over autonomous* ; interagit avec ADR-0006 (Shipper/Merge restent des nœuds first-class — le lieu de l'autonomie) et ADR-0001 (*Sharp tool* — l'idempotence est à l'utilisateur).

## Addendum (2026-07-18, #372) — Ré-activer un Trigger recalcule `next_fire_at` vers l'avant

Ré-activer un Trigger (`PATCH {enabled: true}`) recalcule `next_fire_at` vers l'avant depuis *now*, sans rattraper le slot manqué pendant la désactivation. Un Trigger dormant depuis des jours ne fire donc pas d'un coup à sa ré-activation : il repart au prochain slot cron à venir. Statu quo avant #372 : `set_enabled` ne touchait pas `next_fire_at`, le slot devenu passé était vu comme *dû* au tick suivant et firait aussitôt, un rattrapage implicite né d'une omission que cette décision supprime.

À rapprocher du fire manuel (« Run now », ADR-0027) : un fire manuel *laisse `next_fire_at` intact* (le heartbeat cron possède le planning) ; la ré-activation, elle, *réécrit* `next_fire_at` vers l'avant. Dans les deux cas PDO ne traite jamais un `next_fire_at` périmé comme une raison de firer.

Asymétrie assumée et temporaire : deux chemins conservent volontairement le rattrapage. (1) Le boot/redémarrage du daemon : le premier tick du scheduler (un `tokio::time::interval` dont le premier tick est immédiat) voit tout Trigger activé dont le slot est passé et le fire *une fois* (rattrapage du slot courant, jamais un backfill des slots antérieurs, cf. *forward-only*). (2) La dé-pause globale de #348, non encore implémentée. La divergence est délibérée (une ré-activation manuelle par-Trigger est un « redémarrage propre » ; le boot, un « on reprend où on s'était arrêté ») et documentée comme provisoire : une passe d'unification ultérieure tranchera s'il faut aligner boot/#348 sur le forward-only. N'entame pas l'invariant `next_fire_at` = UTC canonique.

Couture d'implémentation (#372) : un unique stringifieur `CronSchedule::next_fire_utc(now: DateTime<Utc>)` (seul point où `next_fire_at` est stringifié, ce qui rend le `…Z` canonique impossible à violer au niveau *type*) et une unique matrice `recompute_next_fire(now, Transition)` où chaque transition de cycle de vie (`Create`/`CronEdit`/`Repoint`/`Enable`/`CronTick`/`ManualFire`/`Dangling`) choisit *explicitement* son comportement (avancer / laisser / effacer). Le `match` exhaustif force toute transition future à trancher — plus d'omission silencieuse comme celle qui causait le rattrapage sur ré-activation. L'écriture enable + `next_fire_at` passe par un seul `UPDATE` atomique (`UpdateTrigger`).

## Addendum (2026-07-19, #348) — Pause globale des Triggers (kill-switch daemon-wide)

Un flag `triggers_paused` sur `instance_config` court-circuite le tick du scheduler par un early-return *après* le heartbeat de liveness, *avant* `due_triggers`. La pause est une **suppression start-only**, de la même classe que le skip de recouvrement et le cap de sessions : le runtime n'initie toujours aucune action durable de lui-même, il en *supprime*. L'invariant (a)/(b) d'ADR-0012 tient.

Le `enabled` par-Trigger n'est **jamais** muté : « restaurer l'état antérieur à la reprise » est donc gratuit (rien à sauvegarder ni restaurer), robuste au redémarrage (le flag est persisté en base), et cohérent cross-clients (broadcast WS `triggers_paused`, aucun poll). Pause globale et `disabled` par-Trigger sont **deux canaux orthogonaux** — un Trigger désactivé avant la pause reste désactivé après.

Résout l'asymétrie que l'addendum #372 avait *réservée* : la **dé-pause** rattrape le slot courant **une fois** (forward-only, jamais de backfill), exactement comme le premier tick au boot du daemon. Aucune ligne de code dédiée et **aucune transition de cycle de vie ajoutée** (pas de `GlobalUnpause` dans `recompute_next_fire`) : pendant la pause `due_triggers` n'est pas appelé, donc `next_fire_at` reste gelé dans le passé ; le tick ordinaire qui suit la reprise le voit `≤ now` et fire une fois via l'arm `CronTick` existant. La pause est une **grille daemon-wide**, pas un état du Trigger — toute PR qui toucherait `recompute_next_fire`/l'enum `Transition` pour #348 serait un signal d'erreur de conception. Diverge volontairement de la ré-activation par-Trigger (#372, décision B, sans rattrapage) ; cette asymétrie boot/#348-vs-#372 reste comme documentée (le boot et la dé-pause = « on reprend où on s'était arrêté » ; la ré-activation manuelle = « redémarrage propre »).

« Run now » manuel **fire pendant la pause** (permissif, D3-A, derrière une confirmation UI) : le chemin manuel (`fire_trigger_now` → `fire_one_trigger(Manual)`) est disjoint du tick et ne consulte aucun flag de pause. Le précédent 409-on-disabled d'ADR-0027 ne transfère pas — le remède d'un `disabled` est local (réactiver ce Trigger), celui d'une pause est global (dé-pauser tout, ré-armant tous les Triggers), et le geste manuel est justement l'échappatoire assumée. **Pas de nouvel ADR** : additif et réversible (une colonne nullable `triggers_paused`, hors de la machinerie `stored→env→default` de `/settings`, aucune sémantique de projection touchée).
