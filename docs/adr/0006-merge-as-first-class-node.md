# Le Merge devient un nœud first-class, plus auto-spawné

Première itération (CONTEXT.md "Merge Resolver") : un rôle `code-mutating` built-in, **auto-spawné** par le runtime quand un `git merge` entre branches `code-mutating` parallèles produit un conflit. Toggle `auto_merge_resolver: enabled | disabled` au niveau pipeline. Cette formulation a deux défauts : (1) elle introduit de l'orchestration ambiante (le runtime décide de spawner sans que le designer l'ait demandé), ce qui contredit *Sharp tool* (ADR-0001) — le designer ne *voit* pas la barrière de synchro dans son DAG ; (2) elle ne couvre que le cas conflit, pas la barrière elle-même : un fan-out parallèle de N branches `code-mutating` n'a aujourd'hui aucun nœud explicite pour rassembler, c'est implicite. Le besoin a explosé avec l'arrivée du nœud `ForEach` (parallèle obligatoire), qui rend la barrière + fusion un pattern *de premier ordre*, plus une exception.

**Décision : le `Merge` devient un nœud first-class du DAG, type `code-mutating` toujours, à placer explicitement par le designer.** Sa forme : 1 input port `branches: repeated` (accumule N edges parallèles), 1 output port `merged` (artefact résumé du merge avec frontmatter `conflict_count`, `branches: [...]`, et corps narratif). Sémantique : (a) attend que toutes les branches upstream soient terminées (barrière intrinsèque, gratuite vu que le scheduler n'évalue le node que quand ses upstream sont tous `Completed`) ; (b) forke un sous-worktree depuis la branche du Pipeline Run ; (c) tente `git merge` sur chaque upstream qui a une branche dédiée (= les `code-mutating`) ; (d) si conflit → spawn Claude Code dans le sous-worktree, qui lit les artefacts du Blackboard pour reconstituer les intentions, résout, commit ; (e) si pas de conflit → écrit un summary trivial *"merge auto, no conflict"* + commit, sans LLM. Le toggle `auto_merge_resolver` disparaît. Le Merge auto-spawné disparaît.

**Pourquoi.** Choisi contre l'alternative *"garder l'auto-spawn et y greffer les barrières ForEach"* parce que ça aurait empilé deux mécanismes implicites (barrière + résoluteur) que le designer ne voit ni ne configure — incompatible avec *Sharp tool*. Choisi contre l'alternative *"deux nœuds distincts `Merge` (code) et `Join` (doc-only)"* parce que la sémantique "rassembler N branches" est unique du point de vue designer : il wire ses N branches vers le port `branches` et le runtime fait le bon choix selon le type des upstream — forcer deux nœuds pour le même geste est friction inutile. Choisi contre l'alternative *"spawn LLM systématique pour traçabilité"* parce que le coût (un Claude Code par convergence, même triviale) écrase le bénéfice ; le summary trivial sans LLM suffit pour l'audit. Conséquences : pipelines existantes qui s'appuyaient sur l'auto-spawn cassent — un script de migration `pipeline_migrator` doit insérer un Merge node downstream de chaque fan-out `code-mutating` détecté, sinon le runtime n'a plus rien pour résoudre les conflits et la branche du Run reste cassée. Diagnostic info-only dans l'éditeur si fan-out `code-mutating` sans Merge downstream (cf. ADR-0001 : lint info-only autorisé).

## Addendum (2026-06-05) — barrière edge-centrée, superseded par les edges conditionnels (ADR-0011)

La clause (a) de la décision ci-dessus — *« attend que toutes les branches upstream soient `Completed` (barrière intrinsèque, gratuite vu que le scheduler n'évalue le node que quand ses upstream sont tous `Completed` »* — est **superseded**. Elle était node-centrée : le Merge était déclaré prêt quand *tous* ses producteurs d'edges étaient `Completed`. Avec les edges conditionnels d'ADR-0011 (routage `when:`/`else` en multi-match), une branche peut être **routée ailleurs** et ne jamais spawner : son producteur ne devient jamais `Completed`, et l'ancienne barrière attendait alors **éternellement** une branche qui ne viendra jamais — un **stall silencieux** du Run. (Régression observée par un test couche 5 : `classifier` fan-out `severity=high` → hotfix, `security=true` → security-review, `else` → backlog ; artefact `high + true` → hotfix et security-review firent, backlog `else` est supprimée et ne spawne jamais → le Merge attend backlog pour toujours.)

**Décision.** La barrière du Merge devient **edge-centrée**, comme cas particulier de la règle de convergence générale (pas un traitement spécial du `merge`) :

- Une edge entrante est **résolue** quand elle a soit **firé** (producteur `Completed` + garde satisfaite, ou edge inconditionnelle), soit est **morte** (producteur `Completed` mais l'edge n'a pas firé — `when:` faux, ou `else` avec une sœur du même port qui a matché ; ou bien le producteur est lui-même mort).
- Un nœud est **mort** pour ce Run s'il a des edges entrantes ET qu'elles sont **toutes mortes**. La mort se **propage** de l'amont vers l'aval : un producteur complète → on évalue ses edges sortantes (firées/mortes) → tout nœud aval dont *toutes* les entrantes sont désormais mortes devient mort, y compris un `Merge` dont les `branches` sont toutes mortes, et y compris `End`.
- **Prêt** : un nœud (Merge inclus) est prêt quand **toutes ses edges entrantes sont résolues** (firées ou mortes) et qu'**au moins une a firé** ; il spawne en consommant **uniquement les branches firées**. Le Merge part dès que ses branches vivantes ont complété, ignorant les branches mortes.
- **Merge à zéro branche firée** (toutes mortes) : le Merge est lui-même mort et **sauté**, comme tout nœud tout-mort — **tant que `End` reste atteignable** par un autre chemin vivant.
- **Jamais de stall silencieux** : si la cascade de mort rend `End` inatteignable (plus aucun chemin vivant n'y mène), le Run **halt explicitement** dans un état diagnosticable (`run_halted`, message « unrouted »), jamais figé en silence. C'est le principe déjà posé par ADR-0011 pour l'épuisement de boucle (« exhausted — unrouted »), étendu ici à toute convergence.

**Conséquence sur `branches: repeated`.** Le port accumule désormais les branches **réellement firées** (compte dynamique, pas la liste statique des upstream déclarés). Une branche morte ne produit ni artefact ni sous-worktree/branche git, donc elle est **naturellement** exclue du `git merge` et du frontmatter `branches: [...]` du `merged.md` : aucune énumération statique d'upstream n'est faite, chaque nœud qui a réellement tourné est mergé. Le `merged.md` reflète ce qui a convergé.

**Périmètre.** Cette barrière edge-centrée couvre la convergence **hors boucle**. La convergence **par-lap dans une boucle bornée/collection** (état d'edge/nœud par itération) est **différée** (slices boucles #148/#151) ; l'état de résolution est porté par-Run et reste compatible avec une future clé par-itération.

**Pourquoi.** Contre *garder la barrière node-centrée et exempter manuellement les branches `else`* : la liste des branches « optionnelles » dépend de l'artefact runtime, pas du graphe statique — impossible à fixer à l'édition. La résolution edge-par-edge est la seule formulation qui reste correcte quel que soit le routage conditionnel observé. Contre *laisser le Run figé et compter sur un timeout* : un stall silencieux est indiagnosticable et viole le principe « jamais de stall silencieux » d'ADR-0011 ; un halt explicite « unrouted » est routable par le Pipeline Manager.
